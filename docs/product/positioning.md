# Positioning

YAI makes AI-enabled systems controllable and reconstructable without replacing
their stack.

## 1. Primary Category

```text
AI Operational Control Core
```

Long form:

```text
local operational control and memory core for AI-enabled systems
```

YAI is not another intelligence layer. YAI is the operational control layer
around intelligence.

## 2. One-Line Pitch

```text
YAI controls what AI-enabled systems attempt to do, records what happened,
and turns operational residue into memory.
```

Short commercial form:

```text
YAI makes AI-enabled systems controllable and reconstructable without replacing
their stack.
```

## 3. Buyer Pain

The first buyer already has or expects AI-enabled systems that act on real
systems:

```text
agents editing code
tools calling APIs
workflows mutating infrastructure
models invoking external tools
scripts producing operational effects
```

Their pain is not lack of another assistant. Their pain is:

```text
we cannot see every attempt
we cannot reliably gate risky operations
we do not have receipts
we cannot reconstruct what happened
we cannot build operational memory across tools
we do not want to replace our stack
```

## 4. Technical Proof

The product proof is the core loop:

```text
open case
bind subject
submit op attempt
materialize policy gate
emit decision
execute or observe effect
record receipt
append store record
derive graph and memory
serve projection
reconcile gaps
```

If this loop works locally, later product layers can consume it without owning
truth.

## 5. First Wedge

The first wedge is agentic coding / local AI workstation.

Why:

```text
close to current development workflow
easy to observe file/process/model effects
cheap to test locally
demonstrates case-bound subjects
demonstrates policy decisions and receipts
demonstrates operational memory
```

Minimum subjects:

```text
repository
file
directory
process
model
agent
tool
```

Minimum value:

```text
make agent-assisted local work auditable, controllable and reconstructable
```

## 6. Anti-Positioning

Do not sell YAI as:

```text
agent framework
workflow engine
TUI product
assistant
model provider
compliance certificate
cloud dashboard first
```

Correct framing:

```text
not replacing the stack
controlling attempts around the stack
recording receipts
building operational memory
exporting controlled projections
```

Audit readiness is a result of receipts and reconstruction. It is not a promise
of certified compliance.

## 7. Product Ladder

| Step | Product | Buyer | Proof |
|---|---|---|---|
| 1 | YAI Core Developer / Local | developers and AI engineering teams | local attempt -> decision -> receipt loop |
| 2 | YAI Integration Kit | teams with existing agents/tools/workflows | adapters speak core primitives |
| 3 | Operational Memory / Audit Pack | teams not ready for enforcement | observed mode reconstruction and gaps |
| 4 | Professional / Team | teams running repeated AI workflows | multi-case memory, policy packs, exports |
| 5 | Enterprise / Embedded | regulated or infrastructure-heavy teams | custom subject/carrier/policy integration |
| 6 | Platform / Cloud later | organizations | sync and dashboards over core truth |

The platform comes after the local core. It is the GitHub-like layer, not the
first product truth.

## 8. Commercial Boundary

The first value is engineering control and operational memory. The system can
help produce evidence, traceability and audit-ready packets, but it must not
claim certified compliance.

YAI should be described as:

```text
control over attempts
receipts for effects
memory from residue
projection with provenance
reconciliation of gaps
```
